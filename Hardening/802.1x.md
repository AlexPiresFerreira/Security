
# 🔐 802.1X (Port-Based Network Access Control)

> **Tags:** #802.1x #network-security #authentication #radius #eap #nac #wireless #wired

---

## 📌 Visão Geral

**802.1X** é um padrão IEEE para **controle de acesso à rede baseado em porta**, fornecendo autenticação antes de permitir acesso à rede.

🎯 **Objetivo:** Garantir que apenas dispositivos e usuários autorizados possam se conectar à rede (cabeada ou wireless).

**Casos de uso:**
- Redes Wi-Fi corporativas (WPA2/WPA3-Enterprise)
- Redes cabeadas (switches gerenciados)
- VPNs
- Acesso remoto

---

## 🏗️ Componentes do 802.1X

### Arquitetura de 3 Componentes

┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ Supplicant │ ◄─────► │Authenticator│ ◄─────► │ Auth │ │ (Cliente) │ EAP │ (Switch/AP)│ RADIUS │ Server │ └─────────────┘ └─────────────┘ └─────────────┘


---

### 1️⃣ Supplicant (Solicitante)

**Definição:** Dispositivo/usuário que deseja acessar a rede.

**Exemplos:**
- Laptop com Windows/Linux/macOS
- Smartphone (Android/iOS)
- Impressora de rede
- Câmera IP

**Software supplicant:**
- **Windows:** Nativo (Wired AutoConfig / WLAN AutoConfig)
- **Linux:** wpa_supplicant
- **macOS:** Nativo
- **Cisco:** Cisco AnyConnect

---

### 2️⃣ Authenticator (Autenticador)

**Definição:** Dispositivo de rede que **controla o acesso** à porta física ou lógica.

**Exemplos:**
- **Switch gerenciado** (rede cabeada)
- **Access Point** (rede wireless)
- **VPN Gateway**
- **Firewall**

**Função:**
- Bloqueia tráfego até autenticação bem-sucedida
- Encaminha mensagens EAP entre supplicant e authentication server
- Aplica políticas de acesso após autenticação

---

### 3️⃣ Authentication Server (Servidor de Autenticação)

**Definição:** Servidor que **valida credenciais** e decide se concede acesso.

**Protocolo:** RADIUS (Remote Authentication Dial-In User Service)

**Servidores RADIUS comuns:**
- **FreeRADIUS** (open source)
- **Microsoft NPS** (Network Policy Server)
- **Cisco ISE** (Identity Services Engine)
- **Aruba ClearPass**
- **PacketFence**

---

## 🔄 Fluxo de Autenticação 802.1X

### Processo Passo a Passo
```

1. Supplicant conecta à porta (física ou wireless) └─ Porta está em estado "não autorizado"
    
2. Authenticator envia EAP-Request/Identity └─ "Quem é você?"
    
3. Supplicant responde com EAP-Response/Identity └─ "Eu sou joao.silva"
    
4. Authenticator encapsula EAP em RADIUS Access-Request └─ Envia para Authentication Server
    
5. Authentication Server valida credenciais ├─ Consulta Active Directory / LDAP / banco de dados └─ Pode solicitar certificado, senha, etc.
    
6. Authentication Server responde: ├─ RADIUS Access-Accept → Autenticação bem-sucedida ✅ │ └─ Pode incluir VLAN, ACL, bandwidth limit │ └─ RADIUS Access-Reject → Autenticação falhou ❌
    
7. Authenticator recebe resposta: ├─ Se Accept: Porta passa para estado "autorizado" └─ Se Reject: Porta permanece bloqueada
    
8. Supplicant obtém acesso à rede
    

```

## 🔑 Métodos EAP (Extensible Authentication Protocol)

### EAP-TLS (Transport Layer Security)

**Segurança:** ⭐⭐⭐⭐⭐ (Mais seguro)

**Características:**
- Usa **certificados digitais** em ambos os lados (cliente e servidor)
- Autenticação mútua
- Não transmite senhas
- Requer PKI (Public Key Infrastructure)

**Vantagens:**
- ✅ Máxima segurança
- ✅ Resistente a ataques man-in-the-middle
- ✅ Não vulnerável a ataques de dicionário

**Desvantagens:**
- ❌ Complexo de implementar (requer PKI)
- ❌ Gerenciamento de certificados

**Uso:** Ambientes corporativos de alta segurança

---

### PEAP (Protected EAP)

**Segurança:** ⭐⭐⭐⭐

**Características:**
- Cria **túnel TLS** protegido
- Servidor usa certificado, cliente usa senha
- Autenticação em duas fases

**Variantes:**
- **PEAP-MSCHAPv2:** Mais comum (Windows)
- **PEAP-GTC:** Generic Token Card

**Vantagens:**
- ✅ Mais simples que EAP-TLS (não requer certificado no cliente)
- ✅ Senha protegida dentro do túnel TLS
- ✅ Suporte nativo no Windows

**Desvantagens:**
- ❌ Vulnerável se senha for fraca
- ❌ Servidor precisa de certificado válido

**Uso:** Ambientes corporativos (padrão mais comum)

---

### EAP-TTLS (Tunneled TLS)

**Segurança:** ⭐⭐⭐⭐

**Características:**
- Similar ao PEAP
- Mais flexível (suporta mais métodos internos)
- Servidor usa certificado, cliente usa senha ou certificado

**Métodos internos suportados:**
- PAP, CHAP, MS-CHAP, MS-CHAPv2
- EAP-MD5, EAP-GTC, EAP-TLS

**Vantagens:**
- ✅ Flexível
- ✅ Suporta autenticação legada dentro do túnel

**Desvantagens:**
- ❌ Menos suportado nativamente (requer software adicional no Windows)

**Uso:** Ambientes Linux/Unix, universidades

---

### EAP-FAST (Flexible Authentication via Secure Tunneling)

**Segurança:** ⭐⭐⭐

**Características:**
- Desenvolvido pela Cisco
- Usa **PAC (Protected Access Credential)** ao invés de certificados
- Autenticação em duas fases

**Vantagens:**
- ✅ Não requer certificados
- ✅ Rápido de implementar

**Desvantagens:**
- ❌ Proprietário da Cisco
- ❌ Menos seguro que EAP-TLS

**Uso:** Ambientes Cisco

---

### Comparação de Métodos EAP

| Método | Segurança | Certificado Cliente | Certificado Servidor | Complexidade | Uso |
|--------|-----------|---------------------|----------------------|--------------|-----|
| **EAP-TLS** | ⭐⭐⭐⭐⭐ | ✅ Sim | ✅ Sim | Alta | Corporativo (máxima segurança) |
| **PEAP** | ⭐⭐⭐⭐ | ❌ Não (senha) | ✅ Sim | Média | Corporativo (padrão) |
| **EAP-TTLS** | ⭐⭐⭐⭐ | ❌ Não (senha) | ✅ Sim | Média | Universidades, Linux |
| **EAP-FAST** | ⭐⭐⭐ | ❌ Não (PAC) | ❌ Não | Baixa | Cisco |
| **EAP-MD5** | ⭐ | ❌ Não | ❌ Não | Baixa | ❌ Não usar (inseguro) |

## 🔒 Segurança e Boas Práticas

### ✅ Recomendações

#### 1. Usar EAP-TLS ou PEAP
```

❌ Evitar: EAP-MD5 (inseguro) ✅ Usar: EAP-TLS (máxima segurança) ✅ Usar: PEAP-MSCHAPv2 (bom equilíbrio)

```
---

#### 2. Validar Certificado do Servidor
```

Sempre configurar clientes para validar certificado do servidor RADIUS ✅ Verificar CN (Common Name) ✅ Verificar CA (Certificate Authority) ❌ Não aceitar qualquer certificado

```
---

#### 3. Usar Senhas Fortes
```

Mínimo 14 caracteres Complexidade (maiúsculas, minúsculas, números, símbolos) Rotação regular (90 dias)

```
---

#### 4. Segmentação de VLAN
```

Usuários autenticados → VLAN 10 (Corporativo) Convidados → VLAN 99 (Guest) Falha de autenticação → VLAN 999 (Quarentena)

```
---

#### 5. Reautenticação Periódica
```

Configurar timeout de reautenticação (ex: 1 hora) Força renovação de credenciais Detecta dispositivos desconectados

```
---

#### 6. Logs e Auditoria
```

Habilitar logs detalhados no RADIUS Monitorar tentativas de autenticação falhadas Alertar sobre múltiplas falhas (possível ataque)

```
